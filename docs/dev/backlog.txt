===============
Project Backlog
===============

This page tracks planned and possible future work for django-resume. Items here
are not commitments; they are a shared planning list for maintainers.

Completed user-visible work belongs in the :doc:`../changelog`. Backlog items
should be removed from this page when they are released, or rewritten if the
next useful step changes.

Research Needed
---------------

* Add support for `JSON Resume <https://jsonresume.org/>`_.

  Support importing from and/or exporting to the JSON Resume schema. This needs
  research and design work before implementation, especially around field
  mapping, plugin compatibility, validation, and whether django-resume should
  treat JSON Resume as an interchange format or a first-class data model.

  See :doc:`jsonresume`.

* Re-evaluate LLM-assisted plugin generation.

  django-resume still has prompt-based plugin generation helpers, but the old
  MCP server proposal was written for the removed database-backed plugin system.
  Before implementing anything, research what workflow would be useful for the
  current filesystem-based plugin model: a better context generator, an MCP
  server, management commands, documentation improvements, or some combination
  of those.

  Related old issue: `#7 <https://github.com/ephes/django-resume/issues/7>`_.

Editorial Theme — Edit Mode
---------------------------

* Polish the editorial **edit-mode appearance / per-field editing view**.

  The inline edit forms render with the generic default form markup, unstyled in
  the editorial layout — e.g. clicking the identity edit pencil swaps in a plain
  block (name / "Location" / "Social Links" / a small avatar circle) that breaks
  out of the editorial header instead of matching it, so editing fields and the
  photo upload (the avatar circle is the upload trigger) are not obvious. Style
  the editorial edit forms (identity first) so editing is visible and on-brand.
  (Already fixed along the way: the timeline date tab flying off + the entry
  padding collapsing on edit.)

* Make the **"Resources" links editable** (Lebenslauf/CV PDF, Portfolio).

  They are currently hardcoded placeholders in the homepage ``resume_cv.html``
  override (``<a href="#">…</a>``) with no data/plugin backing, so there is no
  edit affordance. Making them editable needs a small data model — a dedicated
  resources/links plugin, or reusing the existing ``projects`` plugin — plus an
  editorial template and an inline edit form. Not a quick fix (plugin-scale).

* Editable section labels (optional/related).

  The rotated script labels ("Contact", "Let's talk about", "Education",
  "Resources") are hardcoded template strings, so they are not editable either.
  Same shape of fix as Resources if they should become data-driven.

Theme Extraction
----------------

* Extract the editorial cover-letter (and the handwriting) from ``homepage`` into
  the django-resume editorial theme. **(Large.)**

  The editorial cover-letter page, its styling, and the self-writing handwriting
  are being developed inside the ``homepage`` project (as template overrides +
  app-local CSS/JS) so the design can iterate fast against a live resume without
  editable-install fragility or package releases. This is deliberate: more
  homepage work lands first (e.g. the portfolio) before any extraction.

  Once the homepage design is finished, frozen, and fully editable, lift it into
  django-resume as a clean theme in one deliberate pass:

  - Move the runtime handwriting (font-free composer + ``glyph_data.json`` +
    ``{% handwriting_label %}`` tag + ``handwriting.css``/``.js`` + tests) from
    ``homepage.handwriting`` into django-resume, wired into the editorial
    ``base.html`` so the theme is self-contained (handwriting ``js-hw`` coexists
    with the hairline ``js-lines`` animation).
  - Move the cover-letter templates and CSS (the ``.cover-letter`` block) into the
    django-resume editorial theme; reconcile the homepage overrides against the
    originals and delete them.
  - Fold the homepage ``CoverPlugin`` subclass fields (``closing``,
    ``signature_name``, ``signature_img``) into the upstream ``CoverPlugin`` and
    drop the homepage override app.
  - Remove the ``homepage.handwriting`` app and the editorial overrides from
    ``homepage``; pin/release django-resume accordingly.
  - Gate on **visual parity**: the CV and the cover letter must render identically
    before/after the move (desktop + mobile + print), per the design spec.
  - Repoint the ``handwriting-anim`` pipeline's ``glyph_data.json`` copy target to
    django-resume.

  Design spec (local/untracked):
  ``docs/superpowers/specs/2026-06-22-editorial-cover-letter-and-handwriting-migration-design.md``.
